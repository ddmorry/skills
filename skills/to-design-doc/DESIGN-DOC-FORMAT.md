# Design Doc Format

`docs/design/<feature-slug>.md`。Google スタイルを PRD / ADR との住み分け用に削った構成。該当しないセクションは省略してよい（無理に埋めない）。

## Template

```md
# <機能名> Design Doc

- 状態: draft | current | superseded by <path>
- 関連: PRD #<N> / Epic #<N> / ADR <番号（あれば）>
- 日付: YYYY-MM-DD

## Context and scope

この機能が何であるかの短い前提と、この doc が扱う範囲・扱わない範囲。
「なぜ作るか」は PRD への参照で済ませ、再掲しない。

## Goals / Non-goals

設計として達成すること・意図的にしないこと（箇条書き）。
PRD のプロダクト目標ではなく、設計上の目標（例: 「既存の X シームを再利用し新規シームを作らない」）。

## 全体設計

モジュール構成・責務分担・データフロー・主要インターフェース。
図（Mermaid / ASCII）歓迎。テストのシームをどこに置くかを明示する。
slice 間で共有するインターフェースはここで一度だけ定義する。

## Slice 別設計

### #<issue番号> <slice タイトル>

- 触るモジュールと変更・追加するインターフェースの形（型・シグネチャ・スキーマ）
- エッジケースとエラーハンドリング
- テスト戦略（何を・どのシームで・prior art があれば参照）
- 前後の slice との受け渡し（依存 slice と共有するもの）

（Issue の数だけ繰り返す。番号は実在の Issue 番号と 1:1 対応させる）

## Cross-cutting concerns

横断的関心のうち該当するもののみ: 認証・権限、ログ・可観測性、性能、
マイグレーション、後方互換、セキュリティ。

## Alternatives considered

検討して退けた設計と理由。実在した選択肢のみ。無ければセクションごと省略。
不可逆な決定がここに紛れていたら ADR へ昇格させる。
```

## Rules

- **読者は実装を委譲された下位モデルの Coding Agent。** 会話の文脈を一切持たずにこの doc と Issue だけで実装する前提で、迷いどころ（エッジケース・エラー・境界）を先回りして潰す。
- **decision-rich な断片は歓迎、網羅列挙は禁止。** 型・シグネチャ・状態機械・スキーマは決定を prose より正確に伝えるので載せる。ファイルパスの網羅列挙や動くデモコードはすぐ古くなるので避ける。
- **1機能1本。** slice ごとに doc を分けない（設計整合性が分散する）。機能が doc 1本で扱えない大きさなら、それは機能の切り方への警告と受け取る。
- 用語は対象リポジトリの CONTEXT.md に従う。
