# risk-scan — リスクを横に広げる

ある案件相談・契約レビューで、法務として検討すべき事項を **横断（breadth）** に、**抜け漏れなく（MECE）** 洗い出す。成果物は **リスクポイントリスト**（観点→論点→リスクポイント）。個々のリスクの深掘り・対応策比較はしない（それは [risk-deep-dive] の仕事）。

リスクは **絶対値**（重要度＝影響×蓋然性）だけでなく **相対値** でも捉える。同じ条項でも「この契約類型として普通か、同種サービス・競合・当社雛形・過去案件と比べて有利か不利か」で意味が変わる。各リスクを **reference class**（比較の物差し）と突き合わせ、**標準／有利／不利／業態内在** のどこに位置するかまで判定する。相談元が本当に知りたいのは、この相対の結論であることが多い。

このスキルは助言（法務Review 支援）に徹する。**承認は人間（法務課長）が行い、AI には委ねない**（コパイロット型。用語は `CONTEXT.md`）。参照してよい社内文書は **CorpStack の `status === "official"` のみ**（`CLAUDE.md`）。

## Step 1 — 案件をつかむ

- 対象の案件ディレクトリを特定する。無ければ `matters/README.md` の命名規則で新規作成する。
- `received/`（受領原本）と `context/`（相談経緯）を読む。契約書レビューなら原本のテキストを `work/` に抽出する。
- 当事者・取引構造・与件（相手方の要望）・申請分類（当社/先方フォーマット・継続/新規、`CONTEXT.md`）を **事実関係の要約** として数行で書く。推測は推測と明記する。
- **reference class（比較の物差し）を棚卸す**: まず契約類型を同定し、相対評価に使える比較対象を集める — ①ドメイン知識・実務相場（推測は「推測」と明記）②同種サービス・競合の公開約款（必要なら `/research` で収集）③当社雛形・規程（CorpStack official）④過去の類似案件（`matters/`）。物差しが無い観点は無理に相対判断せず「要検証」とする。
- 完了基準: 何を評価する案件かが、事実の要約として一読で分かる。**契約類型が同定され、reference class（比較の物差し）が棚卸されている。**

## Step 2 — 観点を立てる（横に広げる）

**観点** = 法令・領域などドメイン的な大カテゴリ。[OBSERVATION-POINTS.md](OBSERVATION-POINTS.md) の標準観点カタログを起点に、案件固有の観点を足す。各観点は次の3系統を当てて裏取りする:

- **(a) 社内ルール** — CorpStack official（規程・方針・契約雛形）。`node scripts/corpstack-read.mjs regulation` / `legal` で該当文書を引く。
- **(b) 法令・判例・実務** — 一次資料で確認する。腰を据えた調査は `/research` に委ね、出典 URL 付きで `work/research/` に残す。
- **(c) 案件固有事実** — 取引構造・与件から立ち上がる固有リスク（例: 同一相手との双方向取引、再委託、期またぎ）。
- **(d) ベンチマーク水準** — 各リスクポイントが reference class に対して標準か・外れているかを見る。判断の **根拠レベル** を必ず区別する: **実証**（同業公開約款・当社雛形・過去案件との実突合あり）／**一般論**（ドメイン知識・実務相場に基づく見立て。実突合なし）／**要検証**（物差しが無く相対判断を保留）。

完了基準: カタログの全カテゴリを **該当／非該当で判定** し、非該当には理由を1行添えた（拾わなかったのではなく、当てて落とした状態）。

## Step 3 — 論点とリスクポイントを書き出す

各観点に **論点**（確認・議論すべきイシュー）を、各論点に **リスクポイント**（その論点に潜む法的リスク）を対応させ、次のスキーマの表に落とす:

| # | 観点 | 論点 | リスクポイント | 重要度 | 水準(ベンチ比) | 根拠・出典 | 確認事項 | 深掘り |
|---|------|------|----------------|--------|--------------|-----------|----------|--------|

- **重要度** = 影響 × 蓋然性の見立て（高／中／低）。
- **水準(ベンチ比)** = reference class と比べた相対位置。**標準**（その類型として普通）／**不利↑**（相手方有利＝当社に不利で要注意。重要度を押し上げる）／**有利↓**（当社に有利）／**業態内在**（条項の巧拙では差が出ない、業態そのものに内在するリスク。運用で対処）／**要検証**（物差しが無く保留）。**根拠レベルをセル内に併記する** — 実突合なら出典を（例 `標準(同業公開約款)`）、ドメイン知識のみなら `一般論` を、物差しが無ければ `要検証` を添える（例 `不利↑(要検証)`）。
- **根拠・出典** = 一次資料の条番号・判例・基準、または CorpStack の document ID。出典なきリスクポイントは書かない。
- **確認事項** = 判定に必要だが未確定の事実（事業部・社内に確認すべきこと）。
- **深掘り** = [risk-deep-dive] での深掘り対象候補か（◎必須／○推奨／—不要）。重要度と併せてハンドオフの選定材料になる。

## Step 4 — 抜け漏れ点検

書き終えたら一手戻り、**「拾えていない観点はないか」** を問う。カタログとの突合に加え、案件与件から想起する（相手方業種・規制業種・国際要素・グループ内取引・反社・データ越境など）。曖昧・未確認は握りつぶさず「確認事項」に積む。

**相対評価も点検する**: ①「不利↑」を根拠なく盛っていないか（実突合が無いなら「一般論」か「要検証」に落とす）②「標準」に埋もれた **業態内在リスク** を見落としていないか（条項は普通でも業態ゆえに残るリスクは別枠で拾う）③根拠薄い相対判断（`要検証`）を「標準」と言い切っていないか。

完了基準: 標準カタログの全カテゴリが表または非該当理由で消し込まれ、追加観点の想起が尽きた。

## 出力

`matters/<案件>/work/リスクポイントリスト_YYYYMMDD.md` に、次を Markdown で書く。冒頭に「本メモは法務内部の検討資料であり、最終判断・対外回答は人間の承認を要する」の注記を置く。

1. **総括（相対リスク評価）** — 成果物の冒頭に 3〜5行で結論を先出しする。この契約を reference class と比べ、**①業界標準の範囲内（相対的に問題なし）／②相対的に不利で要注意（どの条項が・どの物差しに対して）／③業態内在で運用対処（条項では消せず運用で対処すべきもの）** の3つに切り分けて言い切る。相対判断には必ず根拠レベル（実証／一般論／要検証）を添える。
2. **事実関係の要約**（Step 1）。
3. **リスクポイントリストの表**（Step 3）。
4. **確認事項の一覧**。

## 次の一歩

リスクポイントリストを人が確認し、深掘り対象（`深掘り` 列◎○・重要度上位）を取捨選択してから [risk-deep-dive] を起動する。このスキルは自動では深掘りに進まない。

## ガードレール

- 参照は CorpStack official のみ。draft/archived は参照しない。
- 承認しない・締結可否を判断しない。洗い出しと整理に徹する。
- リスクポイントには必ず一次資料の出典を付す。未確認は「確認事項」として明示する。
- **相対判断の過大主張を避ける。** 「業界標準」「一般的」等は実質的な断定になりうる。実突合の裏取りがある場合だけ出典付きで言い切り、ドメイン知識に基づく見立ては **「一般論」**、比較の物差しが無いものは **「要検証」** と明示する。
