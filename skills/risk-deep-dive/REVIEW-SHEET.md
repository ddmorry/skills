# 法務レビューシートの型

深掘りの分析ユニットと、統合成果物（法務レビューシート）の節構成。**リーガルレビューの実物**が `matters/20260706_プレイド_循環取引リスク・基本契約更改/work/`（`リーガルレビュー_…md` と `legalreview_…html`）にある。迷ったら現物を参照する。

## 分析ユニット（1 リスクポイント = 1 ユニット）

各リスクポイントを次の順で掘る。**規範と当てはめを分けて書く**のが肝（法的三段論法）。

- **規範** — 根拠法令の条番号・会計/監査基準・判例・実務指針を一次資料で確定。例:「準委任の受任者は報告義務（民法645条・656条）を負い、実費償還は必要と認められる費用（650条）に限られる」。
- **当てはめ** — 規範を本件事実に適用。事実が欠けるなら「〜なら A、〜なら B」と場合分けし、分岐を残す。
- **考察** — 結論とその理由。反対説・境界事例（どこまでが安全でどこから危険か）にも触れる。
- **リスクの整理** — 誰に対する／何の（民事・行政・刑事・レピュテーション・税務）／どの程度（影響の大きさ × 顕在化の蓋然性）／いつ（時間軸）。
- **対応策オプションと比較** — 打ち手を複数案。各案について「効果／コスト・手間／残存リスク／前提条件」を並べ、**推奨案**とその理由を示す。単一解を押し付けない。

## レビューシートの節構成

`work/法務レビューシート_YYYYMMDD.md` を次の順で構成する（プレイド実物と同じ骨格）:

1. **ヘッダ** — 作成日・対象・根拠調査へのパス・注記「本メモは法務内部の検討資料であり、最終判断・対外回答は人間の承認を要する」。
2. **事実関係の要約（前提）** — 当事者・取引構造・与件・時系列。推測は推測と明記。
3. **総括（結論サマリー）** — 番号付きで結論を先出し（各3〜5行）。「最も危険なのは何か」「推奨スキームは何か」を冒頭で言い切る。
4. **論点別の深掘り考察** — 上の分析ユニットを論点ごとに並べる。規範／当てはめの見出しを保つ。
5. **対応策オプション比較表** — 下記の列。案を横断で比較し結論列を持つ。
6. **判断・確認の決定木** — 下記の Phase 形式。フロー次第で結論が変わる分岐を可視化。
7. **確認事項（優先度別）** — 下記の S/A/B。事業部・社内に投げる質問リスト。
8. **専門家相談の要否** — 弁護士・税理士・（設置あれば）会計監査人に何を・いつ相談するか。
9. **出典・リサーチログ** — 一次資料の URL・条番号、`work/research/` へのパス、未確認事項。

## 判断・確認の決定木（形式）

規律の強度や事実の分岐で結論が変わる案件では、Phase 形式のフローを置く:

```
【Phase 0: 規律強度の確定】
Q0-1 … ─ YES → …／NO → …
【Phase 1: 最重要分岐】
Q1 … ─ 連動あり → …／連動なし → Q2 へ
…
```

## 確認事項の優先度

- **S（分岐を決める・今すぐ）** — 回答次第で結論・推奨が変わる事実。
- **A（評価の精度を決める）** — 重要度・金額規模・適用要件の判定に効く事実。
- **B（体制・進め方）** — 相談状況・背景・今後の整備事項。

## HTML / Artifact 化

- 先に `artifact-design` スキルを読む。レビューシートの節をそのままページの見出しに対応させる。
- **self-contained**（外部 CSS/JS/フォント/画像は禁止、すべてインライン）・**ライト/ダーク両対応**・横スクロールは表など幅広要素だけを内側で。
- **匿名化**: 社外・事業部共有の可能性があれば、相手方名を PD社・MS社 等に伏せた版を用意する（要否はユーザーに確認）。
- 出典は本文中に条番号・基準名で明示し、一次資料 URL は末尾にまとめる。
