# AFK Ship — 出荷の委譲手順（メインセッション用）

実際の git ワークフロー（commit → fetch/rebase → push → PR 作成 → squash マージ）は **ship-executor サブエージェント（Sonnet）** が実行する。あなた（メインセッション）の仕事は、executor が必要とする文脈を会話履歴から組み立てて委譲し、結果をユーザーに報告すること。**通常は自分で git 操作をしない。唯一の例外は rebase コンフリクトの解消**（手順4）— どちらのロジックを残すか・どう統合するかという実装判断そのものなので、実装コンテクストを持たない executor には任せず、実装文脈を持つあなたが行う。

## 1. 前提確認

ユーザーの QA 完了が会話から確認できるか判断する:

- 明示の合図がある（「QA OK」「ブラウザテスト OK」「確認した」等）→ そのまま進む。
- ブラウザ QA が該当しない変更（docs / ADR / 設定 / スキルのみ等）→ 「QA 非該当」として理由つきで進む。
- どちらでもない（実装変更なのに QA の言及がない）→ AskUserQuestion で確認してから進む。勝手に「QA 済み」と書かない。

## 2. 要約の組み立て

executor は会話履歴を見られない。以下を会話とセッションの記録から整理する:

1. **何を・なぜ実装したか** — コミットメッセージと PR 本文の材料になる粒度で。設計判断とその理由も。
2. **検証の実施状況** — typecheck / lint / test / build の結果と件数。このセッションで未実施ならその旨を正直に書く（executor が commit 前に回す）。
3. **関連 Issue 番号** — なければ「Issue なし」と明記。
4. **QA 完了の旨** — または QA 非該当の理由。
5. **作業場所** — 対象リポジトリ・ブランチ・worktree の絶対パス。

## 3. 委譲

Agent ツールで `subagent_type: "ship-executor"` を起動し、上記要約をプロンプトとして渡す。フォアグラウンド（`run_in_background: false`）で実行し、完了を待つ。

## 4. コンフリクトで停止したときの解消（発生時のみ・あなたが行う）

executor は rebase コンフリクトを解消しない（種類を問わず）。実装コンテクストを持たない executor が解消すると、テキスト上は単純に見えても意味的に壊すことがあるため、コンフリクトが出たら executor は衝突の全容を報告して安全な状態（rebase 前）に戻して停止する。解消はあなたの担当:

1. executor の報告（衝突ファイル・両側の内容）と自分の実装コンテクストを突き合わせる。対象リポジトリで `git fetch origin main` → `git rebase origin/main` を回してコンフリクトを再現する。
2. 各衝突を実装意図に沿って解消する。どちらのロジックを残すか・どう統合するかは、あなたが持つ実装文脈で判断する。解消後 `git add <解消したファイル>` → `git rebase --continue`。
3. rebase 後のコードは「自分の変更 × main の新コミット」の未検証の組合せ。プロジェクトの検証一式（型チェック・test・lint・build）を回して緑を確認する。数値（test 件数など）は再委譲の材料として控える。
4. クリーン（rebase 完了・push 前・検証緑）にできたら、**push 以降を executor に再委譲する**（手順3に戻る）。再委譲の要約に「rebase でコンフリクトを解消済み・再検証の結果（件数）・push 前」を明記する。executor は fetch し直して push → PR → マージだけを行う。
5. 解消に設計判断が要る／自分の実装コンテクストだけでは決めきれない場合は、ユーザーと相談してから進める。勝手に片側を捨てない。

## 5. 報告と後始末

- executor の最終報告（マージコミット hash・PR URL・閉じた Issue・未解決事項）をユーザーに伝える。
- executor が前提不足・マージ拒否で停止した場合は、報告内容をもとにユーザーと相談して解決し、解決後に改めて要約を組み立て直して再委譲する。
