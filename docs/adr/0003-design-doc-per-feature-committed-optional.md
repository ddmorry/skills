# Design Doc は機能単位1本・リポジトリにコミット・委譲の必須条件にしない

Coding Agent（下位モデル）に上位モデルの設計品質を持ち込むための Design Doc は、**機能（Epic 相当）ごとに1本**、slice（Issue）別の設計セクションを含む形で書き、対象リポジトリの `docs/design/<feature>.md` にコミットする。各 Issue は本文からパスと該当セクションを参照する。作成は `to-design-doc` スキル（to-issues の後にインタビューなしで合成）で行い、**委譲の必須条件にはしない** — オーケストレータは Design Doc の有無どちらでも動く。

## Considered Options

- **Issue ごとに1本** — 委譲コンテキストは最小になるが、slice 間のインターフェース整合性がドキュメント間に分散し、執筆コストが N 倍になる。
- **Issue 本文への埋め込み（Issue 拡張）** — 参照は gh 一発だが、git 履歴に残らず、機能単位1本の方針とも合わない。
- **委譲の必須条件化** — 品質の下限は上がるが、軽微な Issue 群にも上位モデルの執筆コストが発生し、委譲のフットワークが重くなる（コスト削減という目的と矛盾）。

## Consequences

- slice 間の設計整合性が1ドキュメントで担保され、並列する各 Coding Agent が worktree から同じ全体像を直接読める。
- 設計変更（実装中のエスカレーションによる更新を含む）が git 履歴に残る。
- Design Doc は委譲開始前に origin/main に到達している必要がある（未マージならオーケストレータが先に出荷を促す）。
